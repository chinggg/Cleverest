# Overview

This artifact accompanies the FSE '26 paper **"Evaluating LLM-based Regression Test Generation"**. It contains the implementation of **Cleverest**, a feedback-directed zero-shot LLM-based regression test generation technique, together with the full experimental dataset and scripts to reproduce all main results from the paper.

**Repository**: <https://github.com/chinggg/Cleverest>

**Paper preprint**: <https://arxiv.org/abs/2501.11086>

# Artifact Description

Cleverest is a three-component pipeline: a *Prompt Synthesizer*, an *LLM Module*, and an *Execution Analyzer* running in a feedback loop. Given a commit under test, it generates regression test inputs that trigger the bug introduced (or fixed) by the commit.

The artifact includes:

- **Implementation** – Shell scripts (`run.sh`, `b.sh`, `utils.sh`) implementing Cleverest; an `openai` wrapper script for LLM calls.
- **Configuration files** – Per-program `.env` files (`mujs.env`, `libxml2.env`, `poppler.env`, `jerryscript.env`, `z3.env`, `php.env`, `jq.env`, `micropython.env`) specifying commits under test and parameters.
- **Experimental data** – `repdata.tar.xz` (~6 GB uncompressed): 1 500 `SUMMARY_*` result files and 1 500 `exp_*/` folders with full LLM chat logs (`chat_*.log`), generated inputs (`INPUT_*`), and bug-triggering inputs (`TRIGGER_*`).
- **Aggregated results** – `figure/aggregated_results.csv` (6 760 rows) used to produce all tables in the paper.
- **Fuzzing baselines** – `fuzz/` directory with scripts for ClevFuzz and WAFLGo experiments.
- **Supplementary material** – `supp/supplementary.pdf`.
- **Docker environment** – `Dockerfile` for a fully reproducible Ubuntu 22.04 environment.

# Requirements

**Hardware**: Standard x86-64 machine. No GPU required. At least 8 GB RAM recommended for building subject programs.

**Software**:

- Docker (tested with [aflplusplus/aflplusplus:v4.21c](https://hub.docker.com/layers/aflplusplus/aflplusplus/v4.21c/images/sha256-2c445346a9f5c4e321a08c5d3ae77282ca61ad332dd2ddb7683a724f91d0e136), Ubuntu 22.04)
- OpenAI API key (for `gpt-4o` – the default LLM in the paper)

**Estimated time**:

- Basic smoke test (1 commit, 1 LLM call): < 20 seconds
- Full single-program experiment (all commits, all ablations): several hours

# Installation

Build and launch the provided Docker container:

```bash
docker build -t cleverest .
docker run --rm -it --name clever cleverest
docker exec -it clever bash
```

Set your OpenAI API key inside the container (or in the `openai` script):

```bash
export OPENAI_API_KEY=your_api_key
```

# Basic Smoke Test

Run a single LLM query on one bug-introducing commit of MuJS (should complete in < 20 s):

```bash
export conf=mujs1.env
MAX_ITER=1 ./run.sh $conf
```

Expected output includes `"Bug triggered after commit 8c27b12"` and AddressSanitizer output. Two outputs are produced: a `SUMMARY_*` file and an `exp_*/` folder in the current directory.

# Reproducing Main Results

## Build Subject Programs

```bash
export conf=mujs.env   # replace with libxml2.env, poppler.env, etc.
SCENARIO=BIC ./b.sh $conf   # bug-introducing commits (RQ1 bug-finding)
SCENARIO=FIX ./b.sh $conf   # bug-fixing commits (RQ1 bug-reproduction)
```

## RQ1 – Default Cleverest

```bash
./run.sh $conf
# equivalent to:
SCENARIO=BIC GIT_INFO=FULL MAX_ITER=5 LLM_TEMP=0.5 LLM=gpt-4o ./run.sh $conf
```

## RQ2 – Ablation Study

**Prompt Synthesizer variants:**

```bash
GIT_INFO=MSGONLY  ./run.sh $conf
GIT_INFO=DIFFONLY ./run.sh $conf
GENCMD=1          ./run.sh $conf
```

**LLM Module variants:**

```bash
LLM_TEMP=1.0         ./run.sh $conf
LLM=gpt-4o-mini     ./run.sh $conf
LLM=deepseek-r1     ./run.sh $conf
```

**Execution Analyzer variants:**

```bash
NOFEEDBACK=1  ./run.sh $conf
MAX_ITER=10   ./run.sh $conf
```

## RQ3 – Commit Message Expressiveness

Enhanced/reduced commit messages (generated by Gemini 2.5 Flash) are stored in `msg/gemini.yaml`.

```bash
SCENARIO=FIX GIT_INFO=ENHANCED ./run.sh $conf
SCENARIO=FIX GIT_INFO=REDUCED  ./run.sh $conf
```

## RQ1–RQ3 in one shot

`batch.sh` automates all RQ1/RQ2/RQ3 runs with 10 repetitions for a given program:

```bash
./batch.sh mujs.env   # replace with any other .env file
```

It sequentially runs: default BIC/FIX (RQ1), GENCMD, MSGONLY, DIFFONLY, TEMP1.0, gpt-4o-mini, NOFEEDBACK, ITER10 (RQ2), ENHANCED/REDUCED (RQ3), and DeepSeek-R1, each repeated 10 times.

## RQ4 – Comparison to WAFLGo / ClevFuzz

See `fuzz/README.md` for instructions to run WAFLGo and ClevFuzz experiments.

# Tables and Figures

All tables and figures in the paper are generated from the pre-aggregated CSV files in the `figure/` directory using Jupyter notebooks.

## Step 1 – Aggregate raw results into CSV

`figure/gencsv.py` reads all `SUMMARY_*` files from the experiment directories and aggregates them into `figure/aggregated_results.csv` (6 760 rows, one per commit × scenario × repetition × ablation setting).

```bash
# run from repo root; point --input at directory containing SUMMARY_* files
python figure/gencsv.py --input repdata/ --output figure/aggregated_results.csv
```

Each row records: `subject`, `commit`, `iid` (issue ID), `scenario` (BIC/FIX), `git_info`, `max_iter`, `LLM`, `LLM_temp`, `NOFEEDBACK`, `GENCMD`, `score` (0–3), `success` (0/1), `time`, and `final_result` (B/D/R/X).

The score ordinal scale is: **B** (bug triggered) = 3, **D** (output changed) = 2, **R** (commit reached) = 1, **X/N** (none) = 0.

## Step 2 – Generate Table 1 (Subject Programs)

`figure/gentbl_targets.sh` reads the per-program `.env` files and outputs `figure/targets.csv` with columns: software, description, command, issue IDs, BIC and BFC commit hashes, and GitHub URLs.

```bash
cd figure
./gentbl_targets.sh ../mujs.env ../libxml2.env ../poppler.env \
    ../jerryscript.env ../z3.env ../php.env ../jq.env ../micropython.env
```

## Step 3 – Generate Tables 2, 3, 4 and RQ3 analysis (main paper)

Open and run **`figure/result.ipynb`** in Jupyter. The notebook is self-contained and reads `figure/aggregated_results.csv` and `figure/targets.csv`.

| Section in notebook | Paper location | Content |
|---|---|---|
| **Table 2** | §3 RQ1 | Bug-finding and bug-reproduction effectiveness of Cleverest (default config), per subject and commit |
| **Table 3** | §4 RQ2 | Ablation study: scores for MSGONLY, DIFFONLY, GENCMD, TEMP1.0, gpt-4o-mini, DeepSeek-R1, ITER10, NOFEEDBACK |
| **RQ3 block** | §5 RQ3 | Quantitative impact of enhanced/reduced commit messages on bug-reproduction score |
| **Table 5** | §6 RQ4 | Comparison of Cleverest vs. WAFLGo vs. ClevFuzz (bug count, time-to-find) on 50 WAFLGo-runnable subjects |

Run all cells in sequence; LaTeX source for each table is printed to the cell output.

> **Note – Table 4**: Table 4 in the paper (qualitative examples of how commit message intent affects test case generation) is not generated by a script. It was manually curated from the experimental data in `repdata/`.

## Step 4 – Generate RQ3 figure (`rq3.pdf`)

Open and run **`figure/RQ3.ipynb`**. It reads the same `aggregated_results.csv` and produces `figure/rq3.pdf` (Figure 3 in the paper): a per-commit scatter plot of original vs. modified commit message effectiveness scores, comparing ENHANCED and REDUCED configurations for the FIX scenario.

## Step 5 – Run RQ4 fuzzing experiments and collect results

**WAFLGo baseline** uses per-program Dockerfiles in `fuzz/waflgo/`. Copy the top-level `.env` files and `utils.sh` into `fuzz/waflgo/`, then build and run:

```bash
cd fuzz/waflgo
cp ../../*.env ../../utils.sh .
./bdocker.sh mujs.env      # build WAFLGo Docker image for mujs (repeat per program)
./rdocker.sh mujs.env      # run 24-hour WAFLGo fuzzing campaign
./checkwaflgo.sh mujs.env  # check results → waflgo_mujs_withiid.csv
```

Merge all per-program WAFLGo CSVs into `fuzz/merged_waflgo.csv` (used by `result.ipynb`).

`fuzz/checkseeds.sh` additionally checks how close WAFLGo's initial seeds are to triggering bugs, writing `fuzz/seedscheck_*_withiid.csv`:

```bash
cd fuzz
./checkseeds.sh mujs.env   # repeat per program
```

**ClevFuzz** uses Cleverest-generated inputs as initial seeds for greybox fuzzing. First run the default Cleverest experiments (RQ1), then:

```bash
cd fuzz
./batchfuzz.sh mujs.env <experiment-root>   # runs clevfuzz.sh for all mujs experiments
./checkfuzz.sh mujs.env                      # check results → postfuzz_mujs_withiid.csv
```

Merge all per-program ClevFuzz CSVs into `fuzz/merged_postfuzz.csv`.

**Table 5** in `figure/result.ipynb` reads `fuzz/merged_waflgo.csv`, `fuzz/merged_postfuzz.csv`, and `fuzz/seedscheck_*_withiid.csv` to produce the WAFLGo vs. Cleverest vs. ClevFuzz comparison table.

## Step 6 – Supplementary material

Open and run **`supp/result_undirected.ipynb`**. It reads `supp/aggregated_results_undirected.csv` (produced analogously by `gencsv.py` over the undirected-LLM experiments) and generates the tables in `supp/supplementary.pdf`.
